NTLM 和 Kerberos 的 Pass-the-Hash (PTH) 攻击 ​

NTLM 和 Kerberos 的 Pass-the-Hash (PTH) 攻击分别是基于两种身份验证机制的攻击技术，其核心思路是直接利用认证凭据（如哈希值或票据）来绕过密码验证。以下是两者的详细对比和说明：

1. NTLM 的 Pass-the-Hash 攻击 ​

  • 目标协议: NTLM (NT LAN Manager)
  • 攻击对象: NTLM 哈希值 (通常是 NTLMv1 或 NTLMv2 哈希)
  • 攻击流程:
    1. 攻击者获取了目标用户的 NTLM 哈希值（通过抓取内存、LSASS、抓包或离线密码破解等方式）。
    2. 攻击者使用工具（如 Mimikatz、Pass-The-Hash Toolkit 或 Metasploit）直接注入 NTLM 哈希值，而不是输入明文密码。
    3. 模拟认证请求，通过 NTLM 协议成功登录目标系统（如 SMB、RDP 或其他服务）。
  • 特点:
    • 不需要明文密码，仅依赖 NTLM 哈希。
    • 常用于 Windows 系统，特别是没有启用 Kerberos 的场景。
    • 需要有网络上的目标系统支持 NTLM 协议。
  • 工具:
    • Mimikatz
    • Metasploit 模块 (psexec 等)
    • Impacket 工具集 (wmiexec.py, psexec.py)

2. Kerberos 的 Pass-the-Ticket 攻击 ​

  • 目标协议: Kerberos
  • 攻击对象: Kerberos 票据（TGT 或 TGS）
  • 攻击流程:
    1. 攻击者获取了有效的 Kerberos 票据（通过从受害者系统中提取或伪造）。
      • TGT (Ticket Granting Ticket): 用于请求服务票据。
      • TGS (Ticket Granting Service): 用于访问具体服务。
    2. 攻击者将票据加载到当前会话中，伪装为拥有票据的合法用户。
    3. 使用票据向目标服务（如文件共享、数据库或其他应用）请求访问权限。
  • 特点:
    • 不需要用户密码，只需要票据。
    • 更适用于域环境，尤其是启用了 Kerberos 认证的场景。
    • 攻击的持续性较高，因为票据可以在有效期内重复使用。
  • 工具:
    • Mimikatz (kerberos::ptt)
    • Rubeus
    • Impacket 工具集 (getTGT.py, getST.py)

防御措施 ​

  1. 针对 NTLM PTH:
    • 禁用 NTLM（或限制其使用）。
    • 启用 SMB 签名。
    • 在域控制器上强制使用 Kerberos。
    • 使用多因子认证（MFA）。
  2. 针对 Kerberos PTT:
    • 启用 Kerberos 票据的 PAC 验证。
    • 减少 Kerberos 票据的有效期。
    • 定期清除未使用的 TGT。
    • 使用工具（如 Microsoft ATA 或 Defender for Identity）检测异常的票据使用行为。

通过这些措施，可以大幅降低 Pass-the-Hash 和 Pass-the-Ticket 攻击的成功率。

当我们获取到 krbtgt 用户的 htlmhash 后是不是能构造 tgt ​

ChatGPT 说： ​

ChatGPT

是的，当我们获取到 krbtgt 用户的 NTLM 哈希后，就可以利用它来伪造 Kerberos 的 Ticket-Granting Ticket (TGT)，实现攻击。这种攻击通常被称为 Golden Ticket 攻击。

原理解释 ​

  1. krbtgt 用户的作用:
    • 在 Active Directory 域中，krbtgt 用户是 Kerberos 认证服务的核心账户。
    • 它的密码（即 NTLM 哈希）用于对所有 TGT 进行签名和验证。
    • 如果攻击者获得了 krbtgt 的 NTLM 哈希，就能伪造合法的 TGT。
  2. TGT 的构造:
    • TGT 是 Kerberos 协议中由域控制器 (KDC) 签发的票据，用户凭借 TGT 请求访问其他服务。
    • 攻击者使用 krbtgt 的 NTLM 哈希，生成伪造的 TGT，并在其中嵌入任意用户的身份信息和权限（例如，域管理员权限）。
    • 域控制器会认为这个伪造的 TGT 是合法的，因为它使用了正确的 krbtgt 密钥签名。
  3. Golden Ticket 的能力:
    • 完全控制域：伪造任意用户的 TGT，获取最高权限（如域管理员权限）。
    • 持久化访问：只要 krbtgt 密钥未改变，攻击者的伪造票据就永远有效。
    • 隐蔽性强：TGT 的使用通常不被直接记录，传统日志分析难以检测。

构造 TGT 的方法 ​

利用工具（如 Mimikatz），可以轻松伪造 TGT。

使用 Mimikatz 构造 TGT ​

以下是伪造 Golden Ticket 的步骤：

  1. 获取 krbtgt 用户的 NTLM 哈希：

    • 可以通过工具（如 lsass 或 secretsdump.py）提取。

  2. 使用 Mimikatz 生成 Golden Ticket：

    kerberos::golden /user:Administrator /domain:example.com /sid:S-1-5-21-XXXXX /krbtgt:<krbtgt哈希> /id:500

    参数说明：

    • /user：伪造票据中包含的用户名。
    • /domain：目标域名。
    • /sid：域的安全标识符 (SID)，可以通过 whoami /user 获取。
    • /krbtgt：krbtgt 用户的 NTLM 哈希。
    • /id：用户的 RID，500 是默认的域管理员 RID。

  3. 加载伪造的 TGT：

    kerberos::ptt <ticket_file>

  4. 使用票据访问目标服务：

    • 可以直接使用 dir \\target_server\share 或其他 Kerberos 认证的服务。

防御措施 ​

  1. 定期更改 krbtgt 密码：
    • 更改 krbtgt 用户密码会使所有现有票据失效。
    • 建议每隔 6 个月更改一次，至少更改两次以完全刷新密钥。
  2. 启用认证日志监控：
    • 检测异常的票据使用行为（如非正常时间的管理员登录）。
  3. 实施更严格的访问控制：
    • 限制对敏感账户（如域管理员和 krbtgt）的访问。
    • 使用分层的管理员账户（如 JEA/PAW 模型）。
  4. 增强票据保护：
    • 启用 Kerberos 的 AES 加密（在域控制器和客户端上）。
    • 使用 Microsoft Defender for Identity 等工具检测异常的 Kerberos 行为。

总结 ​

获取 krbtgt 的 NTLM 哈希后，确实可以伪造 TGT 并实现 Golden Ticket 攻击。但这种攻击同时伴随着强烈的风险信号，建议企业采取严格的密钥管理和监控措施来降低风险。

在 Kerberos 协议中，Pass-the-Hash (PTH) 攻击不能直接像 NTLM 一样利用用户的 NTLM 哈希进行身份验证。这是因为 Kerberos 的身份验证机制并不直接依赖用户的 NTLM 哈希，而是基于加密密钥和票据进行操作。然而，攻击者可以通过一些方法间接利用用户的 NTLM 哈希，伪造 Kerberos 票据（类似 PTH 的效果）。以下是详细说明：

Kerberos 是否支持 PTH? ​

严格来说，Kerberos 本身不支持直接的 PTH 攻击，但攻击者可以通过以下方式利用用户的 NTLM 哈希：

1. 利用用户的 NTLM 哈希生成 Kerberos TGT ​

  • 原理：

    Kerberos 票据（如 TGT）的加密依赖用户的 Kerberos 密钥，而该密钥可以从用户的 NTLM 哈希派生。

    • Kerberos 密钥是用户 NTLM 哈希的一个派生值（通过特定算法生成，如 RC4-HMAC）。
    • 如果攻击者拥有用户的 NTLM 哈希，就可以伪造 TGT 并伪装成该用户。

  • 流程：

    1. 获取用户的 NTLM 哈希（通过抓包、LSASS 内存提取等方式）。
    2. 使用工具（如 Rubeus 或 Impacket）生成与该用户匹配的 TGT。
    3. 加载伪造的 TGT 并请求访问目标服务（如文件共享、数据库等）。

  • 工具：

    • Rubeus：这会生成一个与目标用户关联的 TGT，可以进一步用于服务访问。

      Rubeus asktgt /user:username /rc4:<用户的NTLM哈希>

2. Kerberos TGT/TGS 票据的直接利用 (PTH 的变体) ​

  • 如果攻击者获取了用户的 TGT 或 TGS，可以直接加载并使用这些票据进行身份认证，而无需用户密码或 NTLM 哈希。
  • 这种攻击通常称为 Pass-the-Ticket (PTT)，而不是传统意义上的 PTH。
  • 工具：

    • Mimikatz：

      kerberos::ptt <ticket_file>

    • Rubeus：

      Rubeus ptt /ticket:<ticket_file>

PTH 和 Kerberos 的结合 ​

在某些情况下，PTH 和 Kerberos 攻击可以结合起来实现：

1. NTLM 到 Kerberos 的转换 ​

  • 攻击者利用用户的 NTLM 哈希登录支持 NTLM 的服务（如 SMB 或 LDAP）。
  • 在登录过程中，服务端自动生成 TGT 并将其存储在本地内存中。
  • 攻击者可以提取这些 Kerberos 凭据并执行 PTT 攻击。

2. 伪造 TGT（类似 Golden Ticket 攻击） ​

  • 如果攻击者已经拥有 krbtgt 的 NTLM 哈希，可以直接伪造任意用户的 TGT，而无需该用户的哈希或密码。

总结 ​

Kerberos 本身无法像 NTLM 那样直接通过用户的 NTLM 哈希完成身份验证，但攻击者可以通过以下方法实现类似效果：

  • 利用用户的 NTLM 哈希生成 TGT。
  • 获取并利用现有的 Kerberos 票据（PTT）。
  • 结合 NTLM PTH 攻击转化为 Kerberos 攻击。

这些攻击需要一定的权限和工具支持，因此防御策略应集中于以下方面：

  1. 限制对 NTLM 哈希的获取（如保护 LSASS 进程、禁用 NTLM 协议）。
  2. 实施 Kerberos 安全最佳实践（如启用 AES 加密、限制票据有效期）。
  3. 定期监控用户的认证行为和异常票据使用情况。

    • 攻击者使用工具（如 Mimikatz）的 kerberos::ptt 功能将伪造的 TGT 直接注入到当前会话中，模拟了合法用户通过 Kerberos 登录的过程。

  4. 访问目标服务：
    • 攻击者通过 TGS 访问目标服务，获取权限，甚至可以伪装成域管理员访问任何资源。

  2. 传递票据：

    • 通过 Pass-the-Ticket (PTT) 的技术，将金票加载到内存中。
    • 加载后，操作系统会将其作为合法的票据，用于认证 Kerberos 服务请求。

  3. 模拟合法用户会话：

    • 一旦票据被注入，所有通过 Kerberos 的服务请求都会使用伪造的 TGT。
    • 通过设置合适的权限（如域管理员权限），攻击者可以对目标系统进行全面控制。

金票 vs. 票据传递 ​

关键凭据krbtgt 用户的哈希（伪造 TGT）获取到的现成 TGT 或 TGS
票据生成方式由攻击者伪造 TGT使用已存在的 TGT/TGS

防御措施 ​

  1. 定期更改 krbtgt 密码：
    • 更改两次以彻底刷新所有与旧密钥相关的票据。
    • 避免长期使用同一个 krbtgt 哈希。
  2. 加强票据管理：
    • 启用 Kerberos 票据的 PAC 验证。
    • 减少票据有效期（如 TGT 的默认 10 小时）。
  3. 监控和检测：
    • 使用 SIEM 系统检测异常的 Kerberos 行为（如高权限用户的异常登录）。
    • 部署 Microsoft Defender for Identity，监控域控制器的身份验证请求。
  4. 分权管理：
    • 实施分层的管理员账户策略，降低单一用户被攻击的风险。
    • 限制对 krbtgt 哈希的访问权限。

通过这些措施，可以有效降低金票攻击的成功率，同时提升对 PTT 类攻击的检测能力。

原理 ​

  1. TGT 的作用：
    • TGT 是 Kerberos 认证体系中最核心的票据之一。
    • 它用来向域控制器（KDC）请求服务票据（TGS），然后使用 TGS 访问具体的服务（如文件共享、SQL 数据库、远程桌面等）。
  2. 通过 PTT 使用票据：
    • 如果攻击者能够获得普通用户的 TGT，并将其加载到自己的系统中，域控制器会认为这是该用户的合法请求。
    • 所有通过该 TGT 请求的服务都将受到用户权限的限制，攻击者无法超越该普通用户的权限。
  3. 限制：
    • 攻击者在使用普通用户的 TGT时，只能访问该用户原本有权限访问的资源。
    • 无法直接提升权限（除非进一步利用漏洞或配置错误）。

步骤 ​

1. 获取普通用户的 TGT ​

攻击者可以通过以下方法获取 TGT：

  • 内存抓取：使用 Mimikatz 或其他工具从 LSASS 进程中提取用户的 TGT。
  • 网络嗅探：在特定条件下（如不启用加密），可以抓取并解析用户的 Kerberos 认证流量。
  • 票据导出：某些已感染的系统可能已经将 TGT 保存在磁盘中。

2. 加载 TGT ​

使用工具（如 Mimikatz）加载 TGT：

kerberos::ptt <ticket_file>

此操作将票据注入到当前会话的内存中，系统会将其作为默认 Kerberos 凭据。

3. 模拟普通用户访问资源 ​

加载 TGT 后，攻击者可以访问普通用户有权限访问的资源，例如：

  • 网络共享：dir \\server\share
  • 数据库查询：连接数据库服务。
  • LDAP 查询：通过 Kerberos 认证的 LDAP 操作。

提升权限的可能性 ​

虽然直接加载普通用户的 TGT无法提升权限，但攻击者可以结合其他技术扩大攻击范围，例如：

  1. 查找配置错误：
    • 检查普通用户是否有意外高权限（如误配置的目录权限或服务权限）。
    • 检查是否存在可利用的服务账户或弱密码。
  2. 横向移动：
    • 使用普通用户权限访问其他机器，并尝试提取更高权限用户的凭据（如域管理员）。
    • 例如，通过普通用户权限连接到有管理员登录过的机器，从内存中提取管理员票据。
  3. Kerberos 委派攻击：
    • 如果普通用户账户被配置了 Kerberos 委派权限，攻击者可以滥用这些权限来请求高权限服务的票据。

防御措施 ​

  1. 保护 TGT：
    • 启用 Windows 的 Credential Guard，防止 LSASS 内存中的 TGT 被提取。
    • 配置系统强制使用 Kerberos 加密传输，防止网络嗅探。
  2. 最小权限原则：
    • 确保普通用户账户只拥有最低限度的权限。
    • 定期审核权限配置，避免误配置导致权限扩大。
  3. 监控票据活动：
    • 部署 SIEM 或类似工具，检测异常的 Kerberos 活动（如 TGT 的非正常使用）。
    • 监控用户账户的登录行为，发现异常的权限使用。
  4. 限制票据生命周期：
    • 减少 TGT 的有效期（默认 10 小时），降低票据泄露的利用窗口。
    • 启用票据续签限制，防止票据被长期滥用。

总结 ​

获取普通用户的 TGT，并通过 PTT 技术加载后，可以完全模拟该用户的权限，但攻击者的能力受到该用户权限的限制。尽管如此，攻击者仍可以通过横向移动或其他漏洞利用提升权限，因此需要结合权限管理和行为监控来降低风险。

在 Kerberos 协议中，PAC（Privilege Attribute Certificate） 的签名并不是在每次数据包交换时都要检查，而是在特定的场景下由服务或域控制器决定是否验证。以下是 PAC 签名的作用、验证机制和具体细节：

PAC 签名的作用 ​

PAC 包含用户的权限信息（如组成员身份和特权），并附带数字签名以防止篡改：

  1. PAC 的组成：
    • 用户的权限信息。
    • 由 KDC（域控制器）生成的签名（由 krbtgt 密钥加密）。
    • 服务密钥签名（由目标服务密钥加密，用于服务验证）。
  2. 签名的目的：
    • 保证 PAC 的完整性，防止攻击者篡改用户权限信息（如提升用户权限）。
    • 在需要时，目标服务可以验证 PAC 是否来自合法的 KDC。

签名验证的机制 ​

签名验证并非每次都执行，而是在以下情况下进行：

1. 初次验证（服务首次访问时） ​

  • 当用户通过 Kerberos 票据访问目标服务时，服务可以选择验证 PAC 的签名：
    • 检查 PAC 是否由可信 KDC 生成（验证 krbtgt 签名）。
    • 检查 PAC 是否完整且未被篡改（验证服务密钥签名）。
  • 默认情况下，大多数服务不会主动验证 PAC 签名，除非配置为强制验证。

2. 域控制器的验证请求 ​

  • 如果目标服务不具备验证 PAC 的能力，它可以通过 KDC_VERIFY_PAC 请求将 PAC 转发回域控制器，由域控制器验证签名并返回结果。

3. 高安全模式下的强制验证 ​

  • 某些情况下（如敏感系统或启用了特定策略），服务或域控制器会强制检查 PAC 签名，确保票据的合法性。

性能优化与实际行为 ​

  1. 默认行为：
    • 在许多实现中，为了性能考虑，PAC 签名验证并不是每次请求都进行。
    • 某些普通服务可能完全跳过 PAC 签名验证。
  2. 安全增强：
    • 可以启用特定策略（如 Windows 的 PAC 验证策略）强制域内服务验证 PAC 签名。
    • 启用后，所有服务会向域控制器请求 PAC 验证，确保票据真实性。
  3. PAC 验证的负载：
    • 如果在域内广泛启用 PAC 验证，可能会增加域控制器的负载，特别是在大型环境中。

PAC 签名与攻击防御 ​

金票攻击与 PAC 签名的关系 ​

  • 金票攻击：攻击者伪造 TGT 时，也会伪造 PAC。由于伪造的 PAC 使用伪造的 krbtgt 签名，只有攻击者拥有合法的 krbtgt 哈希时，PAC 才能通过验证。
  • 防御机制：
    1. 强制服务验证 PAC 签名（提高攻击门槛）。
    2. 定期更改 krbtgt 密钥，刷新伪造票据的有效性。

如何启用 PAC 验证 ​

在 Windows 环境中，可以通过组策略或注册表启用 PAC 验证：

  1. 组策略

    • 路径：计算机配置 > Windows 设置 > 安全设置 > 本地策略 > 安全选项。
    • 设置：启用 PAC 强制验证。

  2. 注册表修改

    修改以下注册表键值强制启用 PAC 验证：

    HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Lsa\Kerberos\Parameters

    添加键值：

    • ValidateKdcPacSignature，设置为 1 强制验证 PAC 签名。

总结 ​

PAC 签名的验证并不是在每次数据包交换中强制执行，而是在用户初次访问服务或特定安全策略下执行验证。

  • 默认情况下，许多服务可能不会主动验证 PAC 签名，以平衡性能和安全性。
  • 在高安全环境中，可以启用强制 PAC 验证策略，增加对伪造票据（如金票）的防御能力。
  • 需要权衡验证的安全性与对域控制器的性能负载影响。

    • PAC 包含用户的权限信息、组信息等内容。
    • 它附带两个签名：
      • KDC 签名：由 krbtgt 的密钥生成，用于证明 PAC 是由域控制器创建的。
      • 服务密钥签名：由目标服务的密钥生成，用于让服务验证票据是否完整。

1. 如果攻击者持有 krbtgt 的哈希： ​

  • 伪造的 TGT 依然有效。
  • 攻击者使用 krbtgt 的哈希生成伪造的 PAC，并正确签名 KDC 部分。即使 PAC 验证被启用，伪造的 TGT 仍然可以通过验证。

2. 如果攻击者不持有 krbtgt 的哈希： ​

  • 攻击者无法正确生成 KDC 签名。
  • PAC 验证开启后，域控制器会检测到签名不正确，拒绝服务。
  • 此时伪造的 TGT将失效，攻击失败。

金票攻击的影响 ​

金票攻击的关键是伪造合法的 TGT，其中包括 PAC 的伪造。启用了 KDC 签名验证后：

  • 如果攻击者拥有 krbtgt 的哈希：
    • 可以生成有效的 PAC 签名，金票攻击依然有效。
  • 如果攻击者没有 krbtgt 的哈希：
    • 无法生成有效的 KDC 签名，金票攻击失效。

防御角度的考虑 ​

  1. 启用 PAC 验证：
    • 在关键服务上强制启用 PAC 的 KDC 签名验证，可以有效防御未经授权的 TGT 和伪造票据。
    • 配置方式：
      • 组策略：启用 PAC 签名验证策略。
      • 注册表：设置 ValidateKdcPacSignature 为 1。
  2. 定期更换 krbtgt 密钥：
    • 定期更换 krbtgt 密钥可以使攻击者的伪造票据失效。
    • 更换两次以确保历史票据无法通过验证。
  3. 监控异常票据：
    • 部署 SIEM 系统，监控异常的 PAC 验证失败日志。
    • 在域控制器日志中查找相关事件（如 4769 或 PAC 验证失败的记录）。

总结 ​

当 PAC 验证启用且攻击者未能获取 krbtgt 的哈希时，伪造的 TGT 将失效，攻击无法继续。如果攻击者已获取 krbtgt 的哈希，那么即使启用了 PAC 签名验证，伪造的 TGT 依然可以有效使用。因此，保护 krbtgt 密钥和启用 PAC 签名验证是防御伪造票据攻击的关键措施。

    • 如果验证通过，KDC 会生成并返回目标服务的 TGS。
  3. 访问目标服务：
    • 攻击者使用从 KDC 获取到的 TGS（即 ST），向目标服务进行认证，并最终访问服务。

  • Kerberos 的安全设计：
    • Kerberos 协议中，TGT 是一个授权证明，但它不能直接用于访问具体服务。
    • 访问每个服务时，需要用 TGT 请求生成专门的 TGS，TGS 中包含服务端的加密信息。
  • 目标服务依赖 TGS 验证：
    • 目标服务仅接受 KDC 签名的 TGS，而不接受直接使用 TGT。
    • 即使 TGT 是伪造的，只要 KDC 认可了 TGT，并生成了 TGS，服务端会信任这个 TGS。

  3. 使用从 KDC 获取的 TGS，与目标服务交互并完成认证。
  4. 最终，使用合法的 TGS 访问目标服务。
